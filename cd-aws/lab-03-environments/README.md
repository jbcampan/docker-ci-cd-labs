# Lab 03 — GitHub Environments: Staging / Production

## Objective

Differentiate staging and production in the CD pipeline. Staging deploys
automatically on every push to `main`; production requires a manual approval
from a designated reviewer. This is the standard enterprise deployment model.

```
test → build & push ECR (once) → deploy-staging (auto) → [approval gate] → deploy-production
```

## What you learn

| Concept | Explanation |
|---|---|
| **GitHub Environments** | An environment scopes two things independently: which secrets a job can access (only secrets attached to that environment are visible) and which protection rules apply before the job starts. Declaring `environment: production` in a job is what triggers the approval gate — not a branch rule. |
| **`needs:` chaining** | `needs: deploy-staging` makes `deploy-production` invisible to the scheduler until `deploy-staging` has exited with success. If staging smoke-tests fail, production is never queued — no approval prompt is even shown. |
| **`environment:` and OIDC token** | When a job declares `environment: staging`, GitHub includes `environment:staging` in the `sub` claim of the OIDC token issued for that run. The Terraform IAM role's trust policy matches on this exact string — a job without the `environment:` key, or with the wrong environment name, gets an `AccessDenied` from STS before it can do anything at all. |
| **Manual approval gate** | Configured in GitHub UI (Settings → Environments → production → protection rules), not in YAML. The workflow pauses at the `deploy-production` job boundary and sends a notification to the required reviewer. The reviewer clicks Approve in GitHub — then the job starts. |
| **One image, two environments** | The `build-and-push` job runs once and outputs `image_uri` (e.g. `…:a3f9c21`). Both `deploy-staging` and `deploy-production` consume `needs.build-and-push.outputs.image_uri` — they never rebuild. What passed staging is byte-for-byte identical to what reaches production. |
| **Three OIDC roles with different scopes** | The push role (no environment claim) can only push to ECR. The staging deploy role (environment:staging claim) can only update the staging ECS service. The production deploy role (environment:production claim) can only update the production ECS service. A leaked token for one role is useless for the others. |
| **Isolated Terraform state** | Three state files under three S3 keys (`shared/`, `staging/`, `production/`). A `terraform destroy` in `staging/` cannot destroy production resources — they are not in its state. This is the most important blast-radius protection available within a single AWS account. |
| **`terraform_remote_state` data source** | Reads another module's outputs as data without taking ownership of those resources. Staging and production read the ECR ARN and OIDC provider ARN from `shared`'s state — the shared resources are never duplicated and never accidentally destroyed by an environment-level apply. |

## Steps

### Step 0 — Bootstrap the S3 state backend (run once, never again)

This creates the S3 bucket and DynamoDB table that will store all Terraform
state for this lab. It uses a local state file itself — that is intentional
(the bucket that holds state cannot be managed by that same state).

```bash
cd cd-aws/lab-03-environments/terraform/bootstrap

# Choose a globally unique bucket name — S3 bucket names are global across
# all AWS accounts. A good pattern: tfstate-<yourname>-docker-cicd-labs
terraform init
terraform apply -auto-approve \
  -var="bucket_name=tfstate-yourname-docker-cicd-labs"

# Note the outputs:
# bucket_name          → used in every backend.tf below
# dynamodb_table_name  → used in every backend.tf below
```

### Step 1 — Fill in the bucket name in every backend.tf

Replace `REPLACE_WITH_BOOTSTRAP_BUCKET_NAME` in these four files with the
actual bucket name from step 0:

```bash
BUCKET="tfstate-yourname-docker-cicd-labs"   # ← your actual bucket name

for f in \
  terraform/shared/backend.tf \
  terraform/staging/backend.tf \
  terraform/production/backend.tf; do
  sed -i "s/REPLACE_WITH_BOOTSTRAP_BUCKET_NAME/${BUCKET}/g" \
    "cd-aws/lab-03-environments/${f}"
done
```

### Step 2 — Deploy the shared module (ECR + OIDC push role)

```bash
cd cd-aws/lab-03-environments/terraform/shared

# Edit shared.tfvars: set github_repo to your "owner/repo" value
vim shared.tfvars

terraform init
terraform plan -var-file=shared.tfvars
terraform apply -auto-approve -var-file=shared.tfvars

# Note the outputs:
terraform output ecr_repository_uri          # → ECR_URI  (same for both envs)
terraform output github_actions_push_role_arn  # → AWS_ROLE_ARN_PUSH
```

### Step 3 — Deploy the staging infrastructure

```bash
cd ../staging

# Edit staging.tfvars: set github_repo and tfstate_bucket
vim staging.tfvars

terraform init
terraform plan -var-file=staging.tfvars
terraform apply -auto-approve -var-file=staging.tfvars

terraform output github_actions_role_arn   # → AWS_ROLE_ARN  (staging env secret)
terraform output ecs_cluster_name          # → ECS_CLUSTER  (staging env secret)
terraform output ecs_service_name          # → ECS_SERVICE  (staging env secret)
terraform output execution_role_arn        # → for task-definition.staging.json
terraform output task_role_arn             # → for task-definition.staging.json
terraform output alb_dns_name              # → staging app URL
```

### Step 4 — Deploy the production infrastructure

```bash
cd ../production

# Edit production.tfvars: set github_repo and tfstate_bucket (same values as staging)
vim production.tfvars

terraform init
terraform plan -var-file=production.tfvars
terraform apply -auto-approve -var-file=production.tfvars

terraform output github_actions_role_arn   # → AWS_ROLE_ARN  (production env secret)
terraform output ecs_cluster_name          # → ECS_CLUSTER  (production env secret)
terraform output ecs_service_name          # → ECS_SERVICE  (production env secret)
terraform output execution_role_arn        # → for task-definition.production.json
terraform output task_role_arn             # → for task-definition.production.json
terraform output alb_dns_name              # → production app URL
```

### Step 5 — Commit the generated task definition files

`terraform apply` in steps 3 and 4 already wrote `task-definition.staging.json`
and `task-definition.production.json` at the lab root — with real ARNs filled
in automatically via `local_file`. No manual editing or `sed` required.

```bash
cd ../../..   # back to lab-03-environments/

# Verify the files exist and contain real ARNs (not placeholders)
cat task-definition.staging.json    | python3 -m json.tool | grep -E "family|Role"
cat task-definition.production.json | python3 -m json.tool | grep -E "family|Role"

# Commit them so the pipeline can read them on every run
git add task-definition.staging.json task-definition.production.json
git commit -m "feat(environments): add terraform-generated task definitions"
```

> **Why commit generated files?** Same reason as Lab 02: the pipeline reads the
> task definition from the repo at deploy time. Versioning it means every
> deployment is reproducible and changes to CPU, memory, or env vars are
> reviewable in pull requests.

### Step 6 — Create GitHub Environments and set secrets

In **Settings → Environments**, create two environments:

#### `staging` environment

No protection rules needed. Add these **Environment secrets**:

| Secret | Value |
|---|---|
| `AWS_ROLE_ARN` | output of `terraform output github_actions_role_arn` in `terraform/staging` |
| `ECS_CLUSTER` | output of `terraform output ecs_cluster_name` in `terraform/staging` |
| `ECS_SERVICE` | output of `terraform output ecs_service_name` in `terraform/staging` |

#### `production` environment

Protection rules to configure:
- **Required reviewers** → add yourself (or a teammate)
- **Wait timer** → 5 minutes (gives you time to validate staging before the prod job starts even after approval)
- **Deployment branches** → Restrict to `main` only

Add these **Environment secrets**:

| Secret | Value |
|---|---|
| `AWS_ROLE_ARN` | output of `terraform output github_actions_role_arn` in `terraform/production` |
| `ECS_CLUSTER` | output of `terraform output ecs_cluster_name` in `terraform/production` |
| `ECS_SERVICE` | output of `terraform output ecs_service_name` in `terraform/production` |

#### Repository-level secrets (Settings → Secrets and variables → Actions)

These are NOT environment secrets — they are shared by all jobs in the repo:

| Secret | Value |
|---|---|
| `AWS_ROLE_ARN_PUSH` | output of `terraform output github_actions_push_role_arn` in `terraform/shared` |
| `ECR_URI` | output of `terraform output ecr_repository_uri` in `terraform/shared` |

> **Why two levels?** Environment secrets are only visible to jobs that declare
> `environment: <name>`. The `build-and-push` job has no `environment:` key
> (it runs before any approval gate), so it cannot see environment secrets.
> `ECR_URI` and `AWS_ROLE_ARN_PUSH` must therefore live at the repo level.
> The deploy roles (`AWS_ROLE_ARN`) live at the environment level because they
> must NOT be visible to any job that hasn't passed the protection rules.

### Step 7 — Bootstrap: push the first image manually

Both ECS services exist but have no image yet. Push one to unblock them:

```bash
REGION="eu-west-3"
ECR_URI=$(cd terraform/shared && terraform output -raw ecr_repository_uri)

aws ecr get-login-password --region "${REGION}" | \
  docker login --username AWS --password-stdin "${ECR_URI}"

docker build -t "${ECR_URI}:latest" .
docker push "${ECR_URI}:latest"

# Force both services to pick up the first image
for ENV in staging production; do
  CLUSTER="ecs-lab03-${ENV}-cluster"
  SERVICE="ecs-lab03-${ENV}-service"

  aws ecs update-service \
    --cluster "${CLUSTER}" \
    --service "${SERVICE}" \
    --force-new-deployment \
    --region "${REGION}"
done

# Wait for both to stabilise
aws ecs wait services-stable \
  --cluster ecs-lab03-staging-cluster \
  --services ecs-lab03-staging-service

aws ecs wait services-stable \
  --cluster ecs-lab03-production-cluster \
  --services ecs-lab03-production-service
```

STAGING_ALB=$(cd terraform/staging && terraform output -raw alb_dns_name | sed 's|http://||')
curl "http://${STAGING_ALB}/env"   # confirms the new APP_VERSION is running

# Watch production deploy once approved

### Step 11 — Destroy all resources

Destroy in reverse order: staging first (cheapest to validate first, fastest
to clean up), then production, then shared (ECR must be emptied first), then
bootstrap last.

```bash
# ── 1. Empty ECR (required before Terraform can delete it) ───────────────────
ECR_REPO=$(cd terraform/shared && terraform output -raw ecr_repository_name 2>/dev/null || echo "ecs-lab03-app")
IMAGES=$(aws ecr list-images --repository-name "${ECR_REPO}" --query 'imageIds' --output json)

if [ "$IMAGES" != "[]" ]; then
  echo "${IMAGES}" | \
  aws ecr batch-delete-image \
    --repository-name "${ECR_REPO}" \
    --image-ids file:///dev/stdin
fi

# ── 2. Destroy staging ────────────────────────────────────────────────────────
cd terraform/staging
terraform destroy -auto-approve -var-file=staging.tfvars
cd ..

# ── 3. Destroy production ─────────────────────────────────────────────────────
cd production
terraform destroy -auto-approve -var-file=production.tfvars
cd ..

# ── 4. Destroy shared (ECR + OIDC role) ──────────────────────────────────────
cd shared
terraform destroy -auto-approve -var-file=shared.tfvars
cd ..

# ── 5. Clean up generated task definition files ───────────────────────────────
cd ../..   # back to lab-03-environments/
rm -f task-definition.staging.json task-definition.production.json

# ── 6. Destroy bootstrap (S3 bucket + DynamoDB) ───────────────────────────────
# The bucket has prevent_destroy = true — remove it first:
cd bootstrap
# Edit main.tf: change `prevent_destroy = true` to `prevent_destroy = false`
# then:
terraform destroy -auto-approve -var="bucket_name=tfstate-yourname-docker-cicd-labs"
```

## Understanding checks

**1. Why does `environment: production` in the job YAML enforce the approval gate?**

The `environment:` key in a workflow job does two things simultaneously. First,
it tells GitHub which set of secrets to make available to that job. Second, it
tells GitHub to evaluate the named environment's protection rules before
allowing the job to run. The protection rules live in GitHub's settings (not
in the YAML), so they cannot be bypassed by editing the workflow file — an
attacker who gains push access to the repo can change the YAML, but cannot
change the protection rules without admin access to the repository settings.

**2. What prevents the production OIDC role from being assumed by the staging job?**

The IAM trust policy for the production role requires that the OIDC token's
`sub` claim equals exactly `repo:<owner>/<repo>:environment:production`. GitHub
only includes `environment:production` in the `sub` claim when the job itself
declared `environment: production`. The staging job declares
`environment: staging`, so its token carries `environment:staging` — which
does not match the production role's trust policy. AWS STS rejects the
`AssumeRoleWithWebIdentity` call before any AWS action is taken.

**3. Why is the image built once and shared, rather than rebuilt per environment?**

Rebuilding the image for each environment introduces risk: the source code,
dependencies, or base image could change between the two builds (especially
for floating tags like `python:3.12-slim`). By building once and passing the
exact image URI (`<registry>:<sha-tag>`) through `needs.build-and-push.outputs.image_uri`,
both jobs deploy the same digest. This is one of the key properties of a
production-grade pipeline: the artifact that passed testing is exactly the
artifact that reaches production.

**4. Why three Terraform modules (shared, staging, production) instead of one?**

Terraform applies are all-or-nothing within a state. If staging and production
lived in the same state, `terraform destroy` would destroy both simultaneously.
Separating them by state means each environment has its own blast radius. The
shared module exists because the ECR repository and OIDC provider are
AWS-account-scoped resources — they must exist exactly once regardless of how
many environments reference them.

**5. What is the difference between a repository secret and an environment secret?**

A repository secret is visible to any job in any workflow in the repo. An
environment secret is only visible to jobs that declare `environment: <name>`.
In this lab, `ECR_URI` and `AWS_ROLE_ARN_PUSH` are repository secrets because
the `build-and-push` job has no `environment:` key. `AWS_ROLE_ARN`, `ECS_CLUSTER`,
and `ECS_SERVICE` are environment secrets because they differ per environment
and must not be accessible from jobs that haven't passed the protection rules.

## Going further — How to migrate to two AWS accounts

With two AWS accounts (the real enterprise pattern), the only changes would be:

1. Add a second AWS provider alias in Terraform with the production account credentials.
2. Create separate OIDC providers in each account (one per account, not one total).
3. Replace the `tfstate_bucket` variable with two separate buckets (one per account).
4. In GitHub Actions, the two `configure-aws-credentials` steps would assume roles
   in different accounts — no YAML structure change required.

The GitHub Environments setup, the `needs:` chain, the protection rules, and
the task definitions are **identical** whether you use one or two accounts.
That is why learning the pipeline structure first (this lab) is the right
order before tackling multi-account setup.
