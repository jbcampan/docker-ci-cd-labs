# Lab 02 — Automated ECS Fargate Deployment

## Objective

Complete the CD pipeline by automatically deploying to ECS Fargate after every
push to `main`. The full pipeline runs:

```
lint/test → build Docker → push ECR → render task def → deploy ECS → wait stable → smoke test
```

Authentication uses **GitHub OIDC** — no long-lived AWS credentials are stored
in GitHub. The `task-definition.json` consumed by the pipeline is **generated
automatically by `terraform apply`** — no manual editing required.

## What you build

| Resource | Description |
|---|---|
| **ECR repository** | Stores Docker images tagged by commit SHA |
| **VPC + 2 public subnets** | Dedicated lab network across two AZs (required by ALB) |
| **Security groups** | ALB → port 80 public; ECS tasks → port 5000 from ALB only |
| **ALB + Target Group** | Routes traffic to Fargate tasks; health-checks `/health` |
| **ECS Execution Role** | Lets the ECS agent pull from ECR and write to CloudWatch |
| **ECS Task Role** | Runtime AWS permissions for the app (empty here, extensible) |
| **GitHub OIDC provider** | Registers GitHub as a trusted identity provider in IAM |
| **GitHub Actions IAM role** | Assumed by the pipeline via OIDC — no static keys |
| **ECS Cluster + Service** | Runs 1 Fargate task with zero-downtime rolling updates |
| **CloudWatch Log Group** | Receives container stdout/stderr via the `awslogs` driver |
| **`task-definition.json`** | Generated by Terraform with real ARNs; versioned in the repo |

## What you learn

| Concept | Explanation |
|---|---|
| **ECS deployment cycle** | New task def revision → rolling update → ALB health check → old task drained |
| **`task-definition.json` role** | Template file read by `amazon-ecs-render-task-definition`. Only the `image` field changes on each deploy; everything else (roles, log config, health check) stays from this file |
| **Why Terraform generates it** | Terraform already knows all the ARNs it just created (IAM roles, ECR URI, log group). Using `local_file` to write the JSON avoids any manual copy-paste of ARNs |
| **`amazon-ecs-render-task-definition`** | Official AWS action: reads the JSON file, swaps the `image` field, strips read-only fields (`revision`, `taskDefinitionArn`…), outputs a ready-to-register file |
| **`amazon-ecs-deploy-task-definition`** | Registers the rendered JSON as a new revision, calls `update-service`, polls until stable — equivalent to `wait services-stable` |
| **Execution role vs task role** | Execution role: used by the ECS **agent** to start the task (ECR pull, CloudWatch). Task role: used by **app code** at runtime (S3, DynamoDB…). Separate = least privilege |
| **GitHub OIDC (keyless auth)** | GitHub generates a short-lived JWT per run. AWS verifies it and issues temporary STS credentials. No static key rotation, no secret leaks |
| **`ignore_changes` in Terraform** | Terraform manages infrastructure; the pipeline manages deployments. `ignore_changes` on `container_definitions` prevents `terraform apply` from overwriting pipeline-created revisions |
| **`assign_public_ip = true`** | Without a NAT gateway (saves ~$32/month), Fargate tasks need a public IP to reach ECR and CloudWatch endpoints |

## Prerequisites

### Local tools

```bash
terraform --version   # >= 1.7
aws --version         # >= 2.0
docker --version      # >= 24
```

### AWS account

IAM user or role with permissions for: `ECR:*`, `ECS:*`, `IAM:*`, `EC2:*`,
`ElasticLoadBalancing:*`, `CloudWatch:*`, `Logs:*`, `STS:*`.

### GitHub Actions secrets

Set in **Settings → Secrets and variables → Actions → Repository secrets**:

| Secret | Value | Source |
|---|---|---|
| `AWS_ROLE_ARN` | GitHub Actions IAM role ARN | `terraform output github_actions_role_arn` |
| `ECR_URI` | Full ECR repository URI | `terraform output ecr_repository_uri` |
| `ECS_CLUSTER` | ECS cluster name | `terraform output ecs_cluster_name` |
| `ECS_SERVICE` | ECS service name | `terraform output ecs_service_name` |

> No `AWS_ACCESS_KEY_ID` or `AWS_SECRET_ACCESS_KEY` — OIDC replaces them entirely.

## Steps

### 1. Configure Terraform variables

```bash
cd cd-aws/lab-02-ecs-deploy/terraform

cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars and set github_repo to your actual repo, e.g.:
# github_repo = "alice/docker-cicd-labs"
```

### 2. Deploy the infrastructure

```bash
terraform init
terraform plan          # review resources and estimated cost
terraform apply -auto-approve
```

Terraform creates ~20 AWS resources **and writes `task-definition.json`** at the
lab root with all real ARNs filled in. Deployment takes 3–5 minutes.

### 3. Commit `task-definition.json`

This file is generated by Terraform and must be committed so the GitHub Actions
pipeline can read it:

```bash
cd ..   # back to lab-02-ecs-deploy/
git add task-definition.json
git commit -m "chore(ecs-deploy): add generated task-definition.json"
```

> You only do this once. After this the pipeline manages it automatically —
> `terraform apply` will regenerate it if infrastructure changes, and the
> pipeline overwrites only the `image` field on each deploy.

### 4. Set GitHub Actions secrets

```bash
cd terraform/
terraform output github_actions_role_arn   # → AWS_ROLE_ARN
terraform output ecr_repository_uri        # → ECR_URI
terraform output ecs_cluster_name          # → ECS_CLUSTER
terraform output ecs_service_name          # → ECS_SERVICE
```

Copy each value into the corresponding GitHub secret.

### 5. Bootstrap — push the first image

The ECS service exists but cannot start without an image in ECR.
Push one manually before triggering the pipeline for the first time:

```bash
cd ..   # back to lab-02-ecs-deploy/
REGION="eu-west-3"
ECR_URI=$(cd terraform && terraform output -raw ecr_repository_uri)

aws ecr get-login-password --region "${REGION}" | \
  docker login --username AWS --password-stdin "${ECR_URI}"

docker build -t "${ECR_URI}:latest" .
docker push "${ECR_URI}:latest"

# Force the service to pick up the new image
aws ecs update-service \
  --cluster ecs-lab02-cluster \
  --service ecs-lab02-service \
  --force-new-deployment \
  --region "${REGION}"

aws ecs wait services-stable \
  --cluster ecs-lab02-cluster \
  --services ecs-lab02-service
```

### 6. Verify the app is running

```bash
ALB=$(cd terraform && terraform output -raw alb_dns_name)

curl "${ALB}/"        # {"message": "Hello from ECS Fargate!", ...}
curl "${ALB}/health"  # {"status": "healthy", ...}
curl "${ALB}/env"     # runtime environment variables
```

### 7. Trigger a pipeline deployment

```bash
# Edit the app — e.g. change the welcome message in app/main.py

git add cd-aws/lab-02-ecs-deploy/
git commit -m "feat(ecs-deploy): update welcome message — rolling update demo"
git push origin main
```

Watch the **Actions** tab on GitHub. The `deploy` job will:
1. Build and push a new image tagged `sha-<7chars>`
2. Run `render-task-definition` to swap the image in `task-definition.json`
3. Register the result as a new ECS task definition revision
4. Call `update-service` and wait until the new task passes the ALB health check

### 8. Observe the rolling update

```bash
# Watch running/pending/desired counts change during the deployment
watch -n 5 'aws ecs describe-services \
  --cluster ecs-lab02-cluster \
  --services ecs-lab02-service \
  --query "services[0].{running:runningCount,pending:pendingCount,desired:desiredCount}" \
  --output table'
```

### 9. Check container logs

```bash
aws logs tail /ecs/ecs-lab02 --follow
# Or in the AWS Console: CloudWatch → Log groups → /ecs/ecs-lab02
```

### 10. Destroy all resources

```bash
cd cd-aws/lab-02-ecs-deploy/terraform

# ECR must be empty before Terraform can delete it
ECR_REPO="ecs-lab02-app"
IMAGES=$(aws ecr list-images --repository-name "${ECR_REPO}" --query 'imageIds' --output json)

if [ "${IMAGES}" != "[]" ]; then
  echo "${IMAGES}" | \
  aws ecr batch-delete-image \
    --repository-name "${ECR_REPO}" \
    --image-ids file:///dev/stdin
fi

terraform destroy -auto-approve
```

## Understanding checks

**1. Why does Terraform generate `task-definition.json` instead of me writing it?**

Because Terraform already knows all the values that need to go in it — the IAM
role ARNs, the ECR URI, the CloudWatch log group name — since it just created
them. Writing those by hand would mean copying ARNs from `terraform output` and
pasting them in the right places, which is error-prone and adds a manual step.
The `local_file` resource in `ecs.tf` does that automatically at the end of
`terraform apply`.

**2. What does the pipeline actually do with `task-definition.json`?**

The `render-task-definition` action reads the file, replaces the `image` field
with the newly pushed URI, and strips fields that ECS does not accept on
`RegisterTaskDefinition` (`revision`, `taskDefinitionArn`, `status`, …).
The `deploy-task-definition` action then registers the result as a new task
definition revision and calls `update-service` with that revision ARN.

**3. Why is `task-definition.json` committed to the repo?**

It makes deployments auditable: a change to CPU, memory, an environment variable,
or the health check appears in a pull request diff, just like any code change.
It also means the pipeline does not depend on the live AWS state at build time.

**4. What happens if the health check fails on the new task?**

With `minimum_healthy_percent = 100`, ECS never terminates the old task until
the new one is healthy. If health checks fail, ECS drains the new task and keeps
the old version serving traffic. The `wait-for-service-stability` flag times out,
the pipeline step fails, and you get an alert — with zero user impact.

**5. Why OIDC instead of access keys?**

Static AWS access keys stored in GitHub secrets can leak via workflow logs,
forked-repo attacks, or compromised runner environments. With OIDC, GitHub
generates a short-lived JWT per run that AWS verifies cryptographically. The
resulting STS credentials expire when the job finishes. There is nothing to
rotate and nothing to leak.
